# 基金自选与盘中估值产品需求文档

> 文档版本：v1.0  
> 产品代号：净值雷达  
> 产品阶段：个人开发者 MVP / 可运行原型  
> 文档状态：可供设计与开发  
> 更新日期：2026-07-04

## 1. 文档目的

本文档定义一款基金自选与盘中估值应用的产品目标、功能范围、数据来源、估值口径、页面交互、接口设计、验收标准和后续演进方向，用于指导个人开发者完成从 Demo 到可部署 MVP 的开发。

首个版本以中国公募基金为主要对象，优先服务“希望在基金公司公布正式净值前，大致了解基金当日表现”的个人用户。产品不提供交易功能，也不将盘中估值包装成正式净值或投资建议。

## 2. 背景与问题

开放式基金通常在交易日收盘后才陆续公布当日净值。对于股票型、偏股混合型等基金，用户白天可以看到指数和个股波动，却难以快速判断自己持有基金的大致涨跌。

现有体验存在以下问题：

1. 用户需要在多个基金平台之间切换，历史净值、自选和持仓信息不在同一视图。
2. 基金正式净值发布有时间差，盘中缺少易理解的参考指标。
3. 基金持仓可能同时包括 A 股和港股，仅计算 A 股会产生明显遗漏。
4. 第三方平台的估值口径不透明，用户难以判断估值覆盖率和可靠程度。
5. 个人开发者如果一开始加入账号、同步、权限和复杂数据仓库，开发成本会快速失控。

因此，第一阶段应聚焦于一个无需登录、输入服务器 IP 即可打开的轻量 Web 应用。

## 3. 产品定位

### 3.1 一句话定义

一款支持基金自选、历史业绩查看，并依据公开持仓和实时行情估算当日涨跌幅的响应式 Web 应用。

### 3.2 核心价值

- 集中：基金基本信息、历史净值和盘中估值在同一页面展示。
- 及时：正式净值公布前提供可解释的盘中参考。
- 透明：展示持仓日期、行情覆盖率、股票仓位假设和主要贡献项。
- 轻量：无需安装 App、无需注册，手机和 PC 浏览器直接访问。

### 3.3 产品原则

1. 估值必须明确标注“预测”或“估算”，不得与正式净值混淆。
2. 当估值不可用时，历史净值和基本信息仍应正常展示。
3. 优先解释数据覆盖和滞后，而不是制造虚假的高精度。
4. MVP 优先保证一条完整链路跑通，不提前建设用户体系。
5. 对免费数据源的失败、限流和格式变化做好降级处理。

## 4. 产品目标与非目标

### 4.1 MVP 目标

- 用户能够输入6位基金代码并验证基金是否存在。
- 用户能够把有效基金加入自选并在本浏览器持久保存。
- 用户能够查看基金名称、类型、最新正式净值及历史单位净值曲线。
- 系统能够读取最近一期公开股票持仓。
- 系统能够匹配 A 股和港股行情，并对港股计入港币兑人民币变化。
- 系统能够计算预测涨跌幅、预测净值、覆盖率和置信度。
- 页面能够适配手机和 PC。
- 应用能够在 Windows 本地和 Linux 服务器运行。

### 4.2 暂不包含

- 用户注册、登录、找回密码和多设备同步。
- 不同用户之间的服务端数据隔离。
- 基金申购、赎回、券商交易或支付。
- 组合收益、持仓成本、盈亏记账。
- 消息推送、邮件或短信提醒。
- 对基金未来表现的预测或买卖建议。
- 管理后台和复杂运营系统。
- iOS、Android 原生客户端。

## 5. 目标用户与使用场景

### 5.1 目标用户

#### 用户 A：基金持有人

- 持有若干股票型或偏股混合型基金。
- 希望盘中快速了解基金可能的当日表现。
- 能理解估值与正式净值之间存在误差。

#### 用户 B：个人开发者

- 希望用低成本验证基金估值产品是否可行。
- 可以自行运行 Python 服务或部署 Linux 服务器。
- 需要清晰的数据边界、算法口径和错误处理方案。

### 5.2 典型场景

1. 用户第一次打开页面，输入 `026211`，验证“平安科技精选混合C”存在并加入自选。
2. 交易日盘中，用户从手机访问服务器 IP，查看自选基金预测涨跌幅。
3. 用户点击基金卡片，查看过去的单位净值走势。
4. 用户展开估值详情，确认持仓报告日期、行情覆盖率及涨跌贡献最大的股票。
5. 数据源暂时异常时，用户仍能看到已经获取的基本信息或历史数据，并获得清晰提示。

## 8. 核心用户流程

### 8.1 添加基金

1. 用户输入6位基金代码。
2. 前端进行格式校验。
3. 后端查询基金目录。
4. 若基金存在，返回名称和类型。
5. 前端写入 `localStorage` 并刷新自选列表。
6. 若已存在，不重复添加，直接打开详情。

### 8.2 查看估值

1. 页面请求基金概览。
2. 后端读取历史净值和最新正式净值。
3. 后端读取最近一期公开持仓。
4. 根据股票代码区分 A 股和港股。
5. 分别获取实时行情；港股同时获取港币兑人民币汇率。
6. 计算预测涨跌、预测净值、覆盖率和贡献项。
7. 前端显示结果、置信度和风险说明。

### 8.3 查看历史业绩

1. 用户点击基金卡片。
2. 前端请求历史净值接口。
3. 页面以折线图展示日期和单位净值。
4. 数据量较大时允许后端或前端按显示宽度采样，但不得改变首尾数据和趋势。

## 10. 估值模型

### 10.1 输入

- 前一已公布交易日的正式单位净值 `NAV₀`。
- 最近一期公开股票持仓集合。
- 每只股票占基金净值比例 `wᵢ`。
- 股票当日涨跌幅 `rᵢ`。
- 估计股票仓位 `E`，MVP 默认 `90%`。
- 港币兑人民币当日涨跌 `r_fx`。

### 10.2 港股人民币口径收益

港股先计算港币价格收益，再合并汇率变化：

```text
r_hk_cny = (1 + r_hk) × (1 + r_fx) - 1
```

示例：港股上涨 `1.636%`，港币兑人民币下跌 `0.1314%`，人民币口径收益约为：

```text
(1 + 1.636%) × (1 - 0.1314%) - 1 ≈ 1.502%
```

### 10.3 已覆盖持仓的加权平均收益

```text
R_covered = Σ(wᵢ × rᵢ) / Σwᵢ
```

### 10.4 基金预测涨跌幅

MVP 假设已披露且已匹配行情的重仓股能够代表整体股票仓位：

```text
R_fund_est = R_covered × E
```

### 10.5 预测单位净值

```text
NAV_est = NAV₀ × (1 + R_fund_est)
```

### 10.6 覆盖率

```text
Coverage = min(Σwᵢ / E, 100%)
```

其中 `Σwᵢ` 为成功匹配行情的持仓权重，`E` 为假设股票仓位。

### 10.7 置信度

| 覆盖率 | 置信度 |
|---|---|
| `Coverage ≥ 65%` | 高 |
| `40% ≤ Coverage < 65%` | 中 |
| `Coverage < 40%` | 低 |

置信度只是数据覆盖标签，不代表预测一定准确。

### 10.8 已知误差来源

1. 季度持仓披露存在明显时滞，基金经理可能已经调仓。
2. 只披露前十大重仓，无法观察完整股票组合。
3. MVP 股票仓位使用默认值，而不是最新真实资产配置。
4. 债券、现金、期货、可转债和其他资产未精细建模。
5. 港股停牌、跨市场交易时间差会造成行情不同步。
6. 估值未扣除当日管理费、托管费、交易费用和申赎影响。
7. QDII、FOF、商品、债券型基金不适合直接使用该模型。

## 11. 数据需求与来源

### 11.1 MVP 数据源

| 数据 | 首选来源 | 备选来源 | 缓存建议 |
|---|---|---|---|
| 基金目录 | 东方财富基金目录脚本 | 其他基金平台/商业源 | 2小时 |
| 历史单位净值 | 东方财富基金页面数据 | AKShare 封装或商业源 | 5分钟 |
| 股票持仓 | 东方财富基金档案 | 基金定期报告解析 | 1小时 |
| A 股行情 | 东方财富行情接口 | 新浪财经 | 15～30秒 |
| 港股行情 | 新浪财经 | 商业行情源 | 15～30秒 |
| 港币兑人民币 | 新浪财经 | 外汇商业源 | 15～30秒 |

### 11.2 AKShare 定位

AKShare 是开源免费的 Python 数据接口库，本身通常不收取使用费用，但其数据多来自公开网页或第三方接口。使用 AKShare 不等于获得底层数据的商业授权，且接口稳定性取决于底层页面。因此：

- 适合个人研究、原型和低成本验证。
- 不应把其稳定性视为正式 SLA。
- 公网发布和商业化前需要重新确认底层数据条款。
- 关键链路应保留缓存、降级和替换数据源的能力。

### 11.3 数据新鲜度展示

页面应区分以下时间：

- 正式净值日期。
- 持仓报告日期。
- 股票行情时间。
- 估值计算时间。
- 汇率行情时间。

禁止只展示一个模糊的“更新时间”，导致用户误以为所有数据均为实时。

```http

```

主要字段：`code`、`name`、`type`、`latest_nav`、`latest_nav_date`、`latest_daily_return_pct`、`history_count`。

### 12.4 历史净值

```http
GET /api/funds/{code}/history
```

```json
{
  "code": "026211",
  "count": 127,
  "points": [
    {"date":"2026-01-02","unit_nav":1.0123,"daily_return_pct":0.31}
  ]
}
```

### 12.5 盘中估值

```http
GET /api/funds/{code}/estimate?equity_exposure=0.90
```

主要字段：

- `status`
- `estimated_change_pct`
- `estimated_nav`
- `base_nav`、`base_nav_date`
- `covered_weight_pct`、`coverage_ratio`
- `confidence`
- `holding_source_code`、`report_label`
- `a_share_count`、`hk_share_count`
- `fx_applied`、`hkd_cny_change_pct`
- `contributions`
- `disclaimer`

估值失败时返回：

```json
{
  "status": "unavailable",
  "code": "026211",
  "reason": "数据源暂不可用",
  "disclaimer": "仍可查看基金基本信息和历史净值。"
}
```

## 14. 页面与交互规范

### 14.1 首页

- 顶部展示产品名称、简短说明和“非投资建议”提示。
- 添加区域在首屏可见。
- 自选卡片优先展示预测涨跌，其次展示正式净值和覆盖率。
- 涨跌显示正负号；颜色采用中国市场习惯：红涨绿跌。

### 14.2 基金详情

- 标题展示基金名称、代码和类型。
- 四个核心指标：预测涨跌、预测净值、最新正式净值、置信度。
- 历史净值图占据主要视觉区域。
- 持仓贡献表包括股票名称、市场/代码、权重、涨跌和贡献。
- 模型说明展示覆盖权重、股票仓位、A/H 股数量和汇率是否生效。

### 14.3 加载与错误状态

- 初次加载卡片使用骨架屏。
- 基金不存在：提示“未找到该基金”。
- 网络失败：提示“数据源暂不可用，请稍后重试”。
- 无持仓：详情正常展示，但估值区域显示不可用原因。
- 部分股票无行情：使用剩余有效持仓估值，并降低覆盖率。
- 代码重复：不报错，不重复添加，直接打开已有基金。

## 15. 非功能需求

### 15.1 性能

- 首页静态 HTML 在局域网内应在1秒内打开。
- 已缓存的概览接口目标响应时间小于500毫秒。
- 外部数据请求应设置10～20秒超时，禁止无限等待。
- 同一种基金数据应缓存，避免每次页面刷新重复抓取。

### 15.2 可用性

- 单个行情源失败不得导致整个 FastAPI 进程退出。
- 估值接口失败不得影响首页和健康检查。
- 服务重启后无需恢复用户数据，因为自选位于浏览器。

### 15.3 兼容性

- 支持近期版本 Chrome、Edge、Safari 和移动浏览器。
- 后端支持 Python 3.9 及以上版本。
- 支持 Windows PowerShell 启动和 Linux Bash 启动。

### 15.4 可维护性

- 路由、数据服务、估值核心和前端静态文件分离。
- 数据源解析函数应有离线样例测试。
- 免费数据接口变更时，仅替换数据服务层，避免修改页面和估值公式。

## 16. 安全、隐私与合规

### 16.1 MVP 隐私边界

- 不收集姓名、手机号、邮箱、身份证或支付信息。
- 不建立用户账号。
- 自选列表仅保存在浏览器本地。
- 服务端日志不应主动记录不必要的完整请求头和敏感信息。

### 16.2 公网部署安全

- 使用普通非 root 用户运行服务。
- FastAPI 仅监听 `127.0.0.1`，由 Nginx 对外提供80/443端口。
- 配置 HTTPS、基础限流和访问日志轮转。
- 定期更新 Python 依赖。
- 不在代码仓库提交密钥或证书私钥。

### 16.3 风险提示

页面固定展示：

> 盘中估值基于最近一期公开持仓和实时行情推算，不代表基金公司正式净值，不构成投资建议。

公开发布或商业化前，应确认基金数据、股票行情、汇率数据的使用授权，并补充隐私政策和服务协议。

## 20. 风险清单与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 免费接口结构变化 | 页面或估值不可用 | 独立解析层、样例测试、主备源 |
| 服务器 IP 被数据源限制 | Linux 部署后请求失败 | 部署前 curl 验证、更换数据源或地区 |
| 持仓披露滞后 | 估值误差大 | 明示报告日期、统计误差、降低置信度 |
| 股票仓位假设不准确 | 系统性高估或低估 | 后续解析定期报告资产配置 |
| 港股与汇率时间不同步 | 港股贡献偏差 | 展示各自行情时间、允许缺失汇率降级 |
| 用户将估值当正式净值 | 误导风险 | 全局风险提示、预测字段命名、正式净值日期 |
| 公网请求量过大 | 免费源限流、服务不稳定 | 缓存、Nginx 限流、后续任务队列 |

## 22. 结论

该产品适合以个人开发者 MVP 的方式实施。第一阶段不建设账号和数据隔离，将复杂度集中在“基金数据获取、跨市场行情匹配、估值解释和稳定降级”四件真正影响产品价值的事情上。

对于原型而言，HTML 页面并不是脱离后端独立运行，而是由 FastAPI 同时提供页面和 API。最终用户只需在浏览器输入服务器 IP；Python 在服务器上负责抓取数据与计算，浏览器负责交互和展示。这一形态兼顾了开发速度、手机与 PC 兼容性，以及后续迁移到正式部署架构的空间。
